# 跨仓库复用基建地图

## 目标

这个文件记录一个跨仓库事实：

- 哪些能力已经证明应复用
- 这些能力应归位到哪里
- 哪些边界不该为了“公共化”而下沉

它不是产品路线图，也不是一次性的调研结论缓存，而是 foundation 侧维护的长期 system of record。

## 放置原则

### 1. `omne-runtime`

放更硬、更窄、尽量无策略的原语和执行边界：

- archive / compression primitives
- host info / target triple primitives
- integrity / digest primitives
- process primitives
- fs primitives
- system package primitives
- execution gateway

对 `omne_foundation` 的 manifest 边界，再补一条明确约束：

- 这类跨仓原语依赖应通过 canonical git source pin 到 `omne-runtime`
- 不应回退成 `../omne-runtime/...` 之类逃出仓库根目录的 sibling path 依赖

### 2. `omne_foundation`

放应用侧可复用 kit、协议和 adapter：

- config input layer
- HTTP outbound foundation
- MCP runtime / config / session kit
- notify
- secret / prompt / text-assets / i18n / structured text
- policy metadata 的共享语义适配层

### 3. 独立 harness / 治理仓

放让 agent 能稳定发现、验证、维护这些能力的共享治理工具：

- `repo-check`
- docs-system skeleton
- hooks / branch / changelog / version gate

配套机械约束：

- `scripts/check-workspace.sh publish-contract` 现在也会拒绝 workspace 包引用仓库根目录外的 path dependency
- 这样可以直接拦住“文档说跨仓复用，manifest 却偷偷绑 sibling path”的边界回退

### 4. 业务或产品仓

保留离开业务语境就失去意义的流程、策略和产品细节：

- 业务 pipeline
- provider quirks
- source selection strategy
- edge routing / country restriction / rate limiting
- agent orchestration

## 当前推荐复用的领域

### `config` -> `omne_foundation/config-kit`

这是当前最明确的共享 config 输入层。

当前已经覆盖：

- JSON / TOML / YAML 的严格格式识别
- bounded / no-follow 配置文件读取
- 单文档业务 schema 的 allowed-format typed parse helper
- 多层 schema config loader、merge 与 explain 支撑

已证明复用的仓库：

- `captions`
- `ditto-llm`
- `omne-agent`
- `mcp-kit`

下一步最该迁入的仓库：

- `db-vfs`

### `prompt` -> 暂不扩张为统一 DSL；按真实复用面拆开

已确认有明确需求的仓库：

- `captions`
- `omne-agent`

潜在后续消费者：

- `ditto-llm` 的业务 prompt / L1 场景

当前结论：

- `captions` 需要的是 prompt bundle identity：instructions、user text、schema、version、fingerprint、cache key
- `omne-agent` 需要的是 instructions layering：base / user / project / mode / skills / snapshot
- `prompt-kit` 当前只覆盖 prompt 目录 bootstrap 与 lazy access，不代表上面两类语义已经共享

收口原则：

- 不建设通用 prompt DSL、模板引擎或远程分发系统
- 原始目录加载继续保持在 `text-assets-kit` / `prompt-kit` 的窄适配边界
- system prompt layering 继续留在 `omne-agent` harness，直到出现第二个稳定消费者
- 只有当 prompt bundle identity 出现第二个稳定消费者时，才在 `omne_foundation` 新增更高层 prompt crate

### `integrity / sha256` -> `omne-runtime/omne-integrity-primitives`

这是标准 runtime 原语，不应继续在业务仓重复实现。

已证明复用的仓库：

- `captions`
- `ditto-llm`
- `omne-agent`
- `toolchain-installer`

下一步最该迁入的仓库：

- `db-vfs`

### `execution boundary` -> `omne-runtime/omne-execution-gateway`

这是命令执行边界，不是业务编排层。

当前主要消费者：

- `omne-agent`

收口原则：

- gateway 负责 request/policy/audit/sandbox 边界
- app 侧只保留审批、事件编排和产品语义

### `MCP runtime` -> `omne_foundation/mcp-kit`

共享 `mcp.json` 配置加载、连接生命周期与 MCP 语义。

当前状态：

- `mcp-kit` 已是 canonical foundation
- `omne-agent` 已复用配置加载，但连接管理仍未完全收口

### `HTTP outbound foundation` -> `omne_foundation/http-kit`

共享通用 HTTP client、bounded body read、URL 校验与 outbound policy。

已证明复用的仓库：

- `captions`
- `toolchain-installer`
- `mcp-kit`
- `notify-kit`

明确不该下沉到这里的内容：

- `ditto-llm` 的 provider transport quirks
- `toolchain-installer` 的 mirror / gateway / canonical 候选顺序
- `toolchain-edge-gateway` 的 Cloudflare Worker 行为

### `GitHub API client foundation` -> `omne_foundation/github-kit`

共享 GitHub REST API 的纯 client 能力，例如 latest release metadata 获取。

当前状态：

- `toolchain-installer` 已复用
- 后续 `notify-kit` 等 GitHub 调用方可按需要继续接入

明确不该下沉到这里的内容：

- 环境变量读取
- 产品专属 user-agent / token policy
- release asset 命名选择
- mirror / gateway / canonical 候选顺序
- 下载、校验和安装执行

### `policy semantic fragment` -> `policy-meta-spec` + `policy-meta`

共享 canonical policy 命名、版本和 schema。

明确不在这个边界里的内容：

- 运行时决策引擎
- 平台 sandbox 实现
- transport API

### `audio input / media preprocessing` -> `omne_foundation/audio-input-kit` + `omne_foundation/audio-media-kit`

`typemic` 引入了新的跨仓复用方向：桌面音频输入、音频资产元数据和转写前处理。

当前判断：

- 音频设备枚举、采集 session、音频帧格式、权限/设备错误和采集事件是应用侧可复用能力，应进入 `omne_foundation`，而不是长期留在 `typemic`
- `AudioInputBackend` / `AudioDeviceId` / `AudioInputConfig` / `CaptureEvent` / `AudioInputErrorKind` 已开始进入 `audio-input-kit`
- `AudioAssetRef` / `AudioAsset` / `AudioPreprocessTarget` / `AudioProcessingBudget` / `AudioProcessingStep` / `AudioPreprocessProvenance` 已开始进入 `audio-media-kit`
- 音频文件有界读取、原子写入、sha256、派生文件安装/替换等底层原语继续复用 `omne-runtime`
- 具体产品什么时候录音、如何提示用户、历史列表如何呈现，留在 `typemic`

社区复用优先级：

- 跨平台音频 I/O 优先评估 CPAL
- WebView 内 MVP 录音可继续使用 MediaRecorder
- 音频解码优先评估 Symphonia
- WAV 输出优先评估 Hound
- 重采样优先评估 Rubato

foundation 应沉淀的不是这些库的简单再导出，而是稳定领域边界：

- `AudioDeviceId`
- `AudioInputConfig`
- `CaptureSession`
- `AudioAsset`
- 转写输入规范
- decode / resample / normalize pipeline contract
- 有界处理预算和 provenance

### `speech transcription` -> `omne_foundation/speech-transcription-kit`

`typemic` 的远程 OpenAI-compatible 转写只是第一个消费者。转写请求、结果、错误和 provider 能力描述应做成共享领域，而不是产品私有 DTO。

当前判断：

- `TranscriptionRequest` / `TranscriptionResult` / `TranscriptionSegment` / `TranscriptionJob` 已开始进入 `speech-transcription-kit`
- `AudioAssetRef` 已归位到 `audio-media-kit`，并由 `speech-transcription-kit` re-export 给转写调用方
- provider registry、provider capability 描述、统一错误和 provider provenance 已开始进入 `speech-transcription-kit`
- HTTP 出站能力继续复用 `http-kit`
- provider token 解析继续复用 `secret-kit`
- provider 选择、自动转写开关、输出行为留在产品仓

明确不下沉：

- 某个产品的提示词文案
- 用户历史 UI
- 产品默认 provider 策略
- 具体 provider quirks runtime，如果它已经属于 `ditto-llm` 这类更合适的 provider 域

### `text postprocess` -> `omne_foundation/text-postprocess-kit`

`typemic` 的转写后润色 / 优化只是第一个消费者。后处理请求、结果、状态、provenance 和错误应做成共享领域，而不是产品私有 DTO。

当前判断：

- `TextPostprocessRequest` / `TextPostprocessResult` / `TextPostprocessJob` 已开始进入 `text-postprocess-kit`
- 后处理模式已覆盖清理转写、精简、正式化和自定义
- provider/model 选择、后处理 provenance、统一错误分类已开始进入 `text-postprocess-kit`
- LLM provider adapter、模型路由和协议兼容继续属于 `ditto-llm`
- prompt 资产管理继续复用 `prompt-kit`

明确不下沉：

- 某个产品的默认润色说明
- 用户历史 UI 和交付策略
- 具体 LLM provider quirks runtime
- secret 解析和网络请求实现

### `local model assets / inference runtime` -> `omne-runtime` + `omne_foundation/model-assets-kit` + `omne_foundation/speech-whisper-kit`

本地 Whisper 一旦进入 `typemic`，下载、校验、安装、发现和执行都不能写成产品私有脚本。

归位判断：

- 下载、归档、sha256、原子替换、进程执行、sidecar 生命周期这类底层能力属于 `omne-runtime`
- 模型 manifest、模型能力、模型目录、模型选择、转写 runtime adapter 属于 `omne_foundation`
- `ModelManifest` / `ModelSource` / `ModelCapability` / `LocalModelRef` / `ModelInstallProgress` 已开始进入 `model-assets-kit`
- `whisper-rs` 本地执行适配、PCM WAV 校验和样本转换已开始进入 `speech-whisper-kit`
- 哪个模型作为默认、何时提示下载、下载 UI 和产品引导留在 `typemic`

社区复用优先级：

- 本地 Whisper 执行当前优先 `whisper-rs` Rust API
- 纯 Rust ML 路径后续评估 Candle Whisper
- Hugging Face 下载可评估 `hf-hub`，但必须包在我们自己的 manifest、校验和安装边界后面
- Tauri sidecar / shell plugin 只作为兼容外部命令接线，执行策略和生命周期不能散落在产品代码里

foundation 应沉淀：

- `ModelManifest`
- `ModelStore`
- `ModelCapability`
- `LocalModelRuntime`
- `WhisperRsRuntimeConfig`
- PCM WAV 读取和 mono `f32` 样本转换
- 下载来源和校验 provenance
- 本地执行错误：模型缺失、格式不兼容、GPU 后端不可用、进程崩溃、超时

### `desktop input trigger / text delivery` -> `omne_foundation/desktop-input-kit`

全局快捷键、语音唤醒、托盘、剪贴板和直接输入既有平台差异，也有可复用契约。

当前判断：

- Tauri 插件接线留在桌面产品层
- `InputTrigger`、`VoiceActivationSettings`、`VoiceActivationEvent`、`TextDeliveryTarget`、权限错误、快捷键冲突、麦克风不可用、唤醒词不可用和直接输入失败模型已开始进入 `desktop-input-kit`
- 唤醒探测预算、冷却、无语音超时和最长听写限制属于可复用契约，不应只写在产品 UI 里
- macOS Accessibility、Windows SendInput / UI Automation、Linux X11/Wayland 差异必须隔离在平台适配层

社区复用优先级：

- Tauri global-shortcut plugin
- Tauri tray API
- Tauri clipboard-manager plugin
- wake word / keyword spotting 可评估 Porcupine、sherpa-onnx KWS、openWakeWord 等 adapter，但必须由 foundation DTO 包住
- 直接输入可评估 enigo，但必须高权限门控

明确不下沉：

- `typemic` 的具体快捷键默认值
- `typemic` 的具体唤醒词默认值和 UI 动画
- 录音按钮布局
- 用户提示文案
- 产品级“复制到剪贴板还是保留草稿”的默认策略

### `repo governance harness` -> 从 `omne-project-init` 提升为正式共享工具

当前事实来源：

- `omne-project-init` 负责生成 `tools/repo-check/`

推荐方向：

- 在需求稳定后提升为正式共享 harness 项目
- 不放进 `omne-runtime` 的 primitives，也不伪装成业务 crate

## 各仓库归位判断

### `captions`

- 保留：Danbooru adapter、caption contract、caption pipeline、PG cache/result/job flow
- 复用：`config-kit`、`http-kit`、`omne-integrity-primitives`、`ditto-core`

### `db-vfs`

- 保留：DB-backed VFS contract、`VfsPolicy` 语义、service auth/rate-limit/audit
- 复用方向：优先接 `config-kit` 和 `omne-integrity-primitives`

### `ditto-llm`

- 保留：provider runtime、provider routing、provider transport quirks
- 复用：config 输入层、完整性原语、shared foundation kits

### `omne-agent`

- 保留：thread/workflow/checkpoint/artifact/subagent orchestration
- 复用方向：继续收口到 `omne-execution-gateway`、`mcp-kit`、`ditto-llm`

### `toolchain-installer`

- 保留：installation orchestration、source acquisition、managed toolchain 语义
- 复用：`http-kit` + runtime primitives

### `toolchain-edge-gateway`

- 保留独立：Cloudflare Worker、country restriction、rate limiting、fixed route whitelist
- 不下沉到 shared foundation / runtime

### `typemic`

- 保留：语音输入产品流程、录音/转写 UI、用户历史呈现、默认设置、产品体验策略
- 复用方向：`config-kit`、`http-kit`、`secret-kit`、`audio-input-kit`、`audio-media-kit`、`speech-transcription-kit`、`speech-whisper-kit`、`text-postprocess-kit`、`model-assets-kit`、`desktop-input-kit`、`ditto-llm`、`omne-integrity-primitives`、`omne-artifact-install-primitives`
- 需要抽取：音频采集边界、音频资产/预处理边界、转写 provider DTO、文本后处理 DTO、模型 manifest/store/runtime、输入触发与文本交付契约
- 不应保留在产品仓：模型下载器、whisper sidecar 生命周期、平台输入注入通用层、系统 keyring adapter、通用音频格式转换 pipeline、LLM provider 兼容层

### `policy-meta-spec`

- 保留独立：canonical spec repo
- 不扩张成产品级 policy engine

## 不该抽象的部分

- `captions` 的通用 caption pipeline
- `db-vfs` 的通用 VFS service framework
- `ditto-llm` 的 provider quirks runtime
- `toolchain-installer` 的通用下载候选策略
- `toolchain-edge-gateway` 的通用 edge worker foundation
- `omne-agent` 的通用 agent orchestration platform

## 文档维护触发器

- 某个能力新增第二个以上稳定消费者时，检查它是否应该进入本地图。
- 某个共享能力被迁仓、重命名或改边界时，更新本文件。
- 某个产品域被判定“不再下沉”时，也要把否定结论显式写进来，避免重复讨论。
