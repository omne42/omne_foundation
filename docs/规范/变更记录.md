# 变更记录规范

这个文件定义 `omne_foundation` 对 changelog 的维护规则。

当前这些规则由 [`../../scripts/pre_commit_check/`](../../scripts/pre_commit_check/) 在 `pre-commit` 阶段机械执行。

## 当前仓库语义

`omne_foundation` 当前按 crate-package directory 处理：

- 根目录本身不是一个对外发布包
- 真正的包位于 `crates/*/Cargo.toml`
- 每个 crate 各自维护自己的 `CHANGELOG.md`
- 根级 `CHANGELOG.md` 不是当前仓库 changelog，也不是当前提交流程里的有效入口

这里要把“共享检查引擎能力”和“当前仓库语义”分开看：

- `scripts/pre_commit_check/` 作为未来可迁移的检查引擎，仍然保留 `root-package repository` 和 `crate-package directory` 两种形态
- `omne_foundation` 当前通过仓库内容自动识别为 `crate-package directory`
- 所以对当前仓库来说，workspace 根如果再声明 `[package]`，会变成 root package 与 crate package 并存，并被 hook 直接拒绝

## 基本要求

- 任何 crate 内部的实际改动，都必须同时更新该 crate 自己的 `CHANGELOG.md`
- root 级脚本、hook、规范文档这类“目录容器”变更，不要求写根级 changelog，因为根目录本身不作为包发布

同时：

- 不能只改代码不改对应 changelog
- 也不能只改 changelog 而没有实际改动

## 应该写到哪里

- `crates/<name>/` 里的改动写到 `crates/<name>/CHANGELOG.md`
- 一次提交改多个 crate，就同时更新多个 crate 的 changelog
- 根级 `CHANGELOG.md` 不能作为当前仓库提交时的 changelog 补位路径

如果提交只改 root 级文档、hook、脚本而不触及任何 crate，本仓库当前不要求补一个“仓库级 changelog”，也不会通过根级 `CHANGELOG.md` 来补位。

## 只能修改 `[Unreleased]`

默认情况下，只允许修改 changelog 中的 `[Unreleased]` 段落。

已发布版本段被视为不可变历史，不能在普通提交里回写或重写。

如果 hook 发现你改动了已发布段落，会直接拒绝提交。

## release 例外

如果你正在做正式 release，需要有意更新已发布版本段，可以显式使用：

```bash
OMNE_ALLOW_CHANGELOG_RELEASE_EDIT=1 git commit ...
```

这不是日常开发默认路径，只有在明确做 release 切分时才应该使用。

## 反例

下面这些提交会被 `pre-commit` 拒绝：

- crate-package directory 下，改了 `crates/i18n-kit/` 但没改 `crates/i18n-kit/CHANGELOG.md`
- 只改 `crates/*/CHANGELOG.md`，没有任何实际变更
- 在当前仓库里尝试把根级 `CHANGELOG.md` 当作本次提交的 changelog 入口
- 修改 `[0.x.y]` 这样的已发布段落，但没有显式 override

## 目的

这套规则的目标是：

- 保证变更记录和实际提交同步出现
- 把“发布说明”前置到开发流程里，而不是事后补写
- 保持已发布历史段落稳定，避免 release 之后的记录漂移
