# 提交与分支规范

这个文件定义 `omne_foundation` 对分支命名和提交消息的要求。

当前这些规则由下面两个入口机械执行：

- [`../../githooks/pre-commit`](../../githooks/pre-commit)
- [`../../githooks/commit-msg`](../../githooks/commit-msg)

## 分支命名

默认允许的分支名只有两类：

- 主分支：`main`、`master`
- 前缀分支：`feat/`、`fix/`、`docs/`、`refactor/`、`perf/`、`test/`、`chore/`、`build/`、`ci/`、`revert/`

例如：

- `feat/structured-text-rendering`
- `fix/i18n-fallback`
- `docs/foundation-definition`
- `chore/release-prep`

下面这些情况会被 hook 直接拒绝：

- 没有前缀的普通功能分支
- 未列入允许集合的自定义前缀
- 把工作直接堆在语义不清的临时分支上

## 提交消息格式

默认要求使用 Conventional Commits：

```text
<type>(<scope>)!: <subject>
```

其中：

- `type` 必须是：`feat`、`fix`、`docs`、`refactor`、`perf`、`test`、`chore`、`build`、`ci`、`revert`
- `scope` 可选，但如果写了，只允许小写字母、数字、`.`、`_`、`-`
- `!` 可选，用于显式声明 breaking change
- `subject` 必须存在

例如：

- `feat(secret): add vault-backed resolver`
- `fix(i18n): normalize unknown locale fallback`
- `docs(repo): describe release workflow`
- `refactor(runtime-assets)!: split bootstrap surface`

## 允许直接通过的特殊提交

下面这些提交消息属于 git 常见特殊形态，hook 会直接放行：

- `Merge ...`
- `Revert "..."`
- `fixup! ...`
- `squash! ...`

## 与 breaking change 的关系

如果某次提交触发了非 `0` 大版本的 major bump，仅仅修改 `Cargo.toml` 还不够。

此时 `commit-msg` hook 还要求：

- 提交消息必须带 `!`
- 提交消息仍然要符合 Conventional Commits 格式

例如：

- `refactor(core)!: start 1.0 transition`

更完整的版本规则见 [`./版本与兼容.md`](./版本与兼容.md)。

## 目的

这些约束的目标不是形式主义，而是：

- 让分支语义和提交意图在仓库内清晰可见
- 让 hook、发布流程和 changelog 规则有稳定输入
- 让人和 agent 都能快速判断某次提交属于什么类别
