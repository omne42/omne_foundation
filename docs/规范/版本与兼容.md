# 版本与兼容规范

这个文件定义 `omne_foundation` 对“版本号”和“兼容性”的解释。

相关规范：

- [`./README.md`](./README.md)
- [`./提交与分支.md`](./提交与分支.md)
- [`./变更记录.md`](./变更记录.md)
- [`./Hook与质量门禁.md`](./Hook与质量门禁.md)

这里的规则默认作用于 `crates/` 下的活跃 crate。`bak/`、已废弃墓碑目录、未对外承诺的内部草稿，不在这个兼容承诺范围内。

`omne_foundation` workspace 根本身不是一个对外发布包。根 `Cargo.toml` 的 `workspace.package.version` 只是在需要时给 `crates/` 下使用 `version.workspace = true` 的 crate 提供默认版本值，不单独作为一个发布目标参与兼容判断。

`scripts/pre_commit_check/` 作为未来可迁移的共享检查引擎，底层仍支持 `root` / `crate` 两种仓库形态；但 `omne_foundation` 当前的 hook wrapper 显式固定为 `crate`。因此，如果 workspace 根声明了 `[package]`，当前仓库提交时仍会因为布局和预期不符而失败。

## 核心原则

- 兼容性不是凭感觉判断，而是先看 crate 自己的版本号。
- 默认禁止随意升级大版本。只有在明确声明“这次就是要做大版本切换”时，才允许升级大版本。
- 兼容判断按 crate 分开进行，不按整个仓库给一个笼统结论。
- 当前 `omne_foundation` 的活跃 crate 处在 `0.*.*` 阶段。
- 当 crate 版本处在 `0.*.*` 时，这个仓库当前不对兼容和版本匹配施加任何强制限制。

## 版本判断单位

`omne_foundation` 的每个 crate 都维护自己的版本，兼容判断也按各自的 `package.version` 进行。

这意味着：

- 兼容判断对象是 `crates/` 下的 crate，不是 workspace 根本身。
- `crates/` 下的版本声明只能选择一种模式：
  - root-level 模式：所有 crate 都使用 `version.workspace = true`
  - crate-level 模式：所有 crate 都显式声明自己的 `package.version`
- 不允许部分 crate 继承 workspace 版本、另一部分 crate 写死显式版本。
- `secret-kit` 的 breaking change 只要求提升 `secret-kit` 自己的版本，不自动要求其他 crate 同步做同级别提升。
- 一次仓库提交可以同时改多个 crate，但每个 crate 仍然要单独判断自己应该升哪个版本段。
- 如果某个 crate 通过 `version.workspace = true` 继承版本，那么兼容判断看的仍然是这个 crate 最终解析出来的版本，而不是把 workspace 根当成一个统一发布包。

对当前仓库的现状，再补一条明确说明：

- 当前 `omne_foundation` 实际采用的是 crate-level 模式，活跃 crate 都显式声明自己的 `package.version`

## 什么算外部接口

下面这些内容，只要已经对外暴露并允许下游依赖，就算兼容判断范围的一部分：

- Rust `pub` API，包括公开类型、函数、trait、macro 和 re-export
- 对外文档已经承诺的行为语义
- CLI 的命令名、子命令、参数、退出语义
- 配置文件格式、环境变量约定、URL/spec 语法
- JSON Schema、线协议 DTO、生成绑定等跨语言边界

下面这些内容默认不算稳定兼容承诺，除非文档另有明确声明：

- 未导出的内部模块和私有实现
- `bak/` 下的历史遗留物
- 明确标记为实验性的草稿接口
- 只服务于测试的内部辅助代码

## `a.b.c` 的含义

### `a`：大版本

`a` 表示大版本。

规则如下：

- 如果没有明确声明“这次需要大版本切换”，禁止更新大版本。
- 当 `a > 0` 时，只有大版本更新才允许破坏面向外部的接口。
- 当 `a > 0` 时，同一个大版本内必须保持向后兼容。

对当前仓库的特殊约束：

- 当前活跃 crate 处在 `0.*.*` 阶段。
- 在 `0.*.*` 阶段，不做兼容性强约束，也不要求外部接口变化必须匹配某个固定版本段。
- 只有在明确决定进入 `1.0.0` 或其他新大版本时，才允许把 `a` 从 `0` 改到非 `0`。

### `b`：小版本

`b` 表示小版本，一般用于一批较大内容的更新。

规则如下：

- 当 `a > 0` 时，提升 `b` 只能做向后兼容的扩展、增强或成批新增能力，不能破坏已有外部接口。

### `c`：补丁版本

`c` 表示轻微改动或紧急修复。

规则如下：

- `c` 只用于小修、小范围行为修正、实现优化或紧急修复。
- `c` 不应该承载新的兼容债，也不应该破坏已有外部接口。
- 无论当前 `a` 是否为 `0`，同一个 `a.b.*` 内的 `c` 更新都应保持向后兼容。

## 发布时怎么判断该升哪一段

可以按下面顺序判断：

1. 先确定这次改动影响了哪些 crate。
2. 对每个 crate 单独判断，是否破坏了它已经承诺的外部接口。
3. 如果当前版本是 `0.b.c`，当前仓库不对这次改动施加版本匹配限制，由维护者自行决定是否调整版本。
4. 如果当前版本是非 `0` 大版本，且这次改动破坏了外部接口，只有在明确批准大版本切换时才升级到新的大版本。
5. 如果当前版本是非 `0` 大版本，且没有破坏外部接口，但属于一批较大的新增或调整，升级 `b`。
6. 如果只是轻微改动或紧急修复，升级 `c`。

## 例子

- `i18n-kit` 在 `0.1.0` 删除一个现有 `pub` 函数并改掉调用方式：当前仓库不会在 `0.*.*` 阶段用 hook 强制要求某个版本段。
- `structured-text-kit` 在 `0.1.0` 新增一组构造 API：当前仓库不会在 `0.*.*` 阶段用 hook 强制要求 bump 小版本。
- `secret-kit` 只是修正一个不改变公开接口的解析 bug：应发布补丁版本。

## 结论

`omne_foundation` 当前对兼容的理解可以压缩成三句话：

- 先看 crate 自己的版本号，再谈兼容。
- 没有明确声明要升大版本时，禁止随意升大版本。
- 当前处在 `0.*.*` 阶段时，不做兼容与版本匹配的强制限制。

## Hook 约束

仓库的 githook 会对这个规则做最低限度的机械约束：

- hook 会自动识别当前提交采用的是 root-level 版本模式还是 crate-level 版本模式。
- 如果 `crates/` 下同时混用 `version.workspace = true` 和显式 `package.version`，hook 会直接报错。
- hook 只检查 `crates/` 下 crate 的有效版本，不把 workspace 根当成一个发布单元。
- 当 crate 当前版本或目标版本仍处在 `0.*.*` 时，hook 不对它施加兼容与版本匹配限制。
- 当 crate 进入非 `0` 大版本后，`pre-commit` 会比对 staged Rust public API 和版本号，拦住“外部 Rust 接口变了，但版本段没有按规则变化”的提交。
- 在非 `0` 大版本阶段，检测到 Rust public API breaking change 时，必须提升大版本 `a`；仅新增/扩展 public API 时，至少要提升小版本 `b`。
- 对非 `0` 大版本的 staged `Cargo.toml` 大版本提升，默认仍然拒绝。
- 如果确实要做非 `0` 大版本切换，必须显式使用 `OMNE_ALLOW_MAJOR_VERSION_BUMP=1 git commit ...`。
- 发生非 `0` 大版本变更时，`commit-msg` 还要求使用带 `!` 的 Conventional Commit，明确声明这是 breaking change。

当前自动化 gate 的机械覆盖范围是 Rust public API。CLI、配置格式、协议语义、运行时行为语义等其他外部接口，仍然需要按本文件规则做工程判断，不能把责任全部推给 hook。
